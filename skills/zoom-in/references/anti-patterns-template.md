# ANTI-PATTERNS.md Template

This template defines the structure of `ANTI-PATTERNS.md` at the project root.
When `/zoom-in init` generates the file, it follows this structure. Unlike the
universal `references/anti-patterns.md` (shipped with the skill), this file is
PROJECT-SPECIFIC: it tailors every universal No-Go with a concrete example from
the actual codebase, and pulls in framework- and register-specific bans detected
during init.

---

> **This file is LIVING.** It grows with every `/zoom-in adopt --ban` and every
> new pattern discovered during `/zoom-in audit`. It is the single source of
> truth for what this project rejects and why.

### 1. Database Queries Inside Loops (N+1)

**Project Example:** [discovered during init — e.g., "Found in `modules/orders/services.py:45`: `for order in orders: customer = Customer.objects.get(id=order.customer_id)`" or "not yet found — will be flagged if detected"]

**Detection:** Any DB access (ORM query, raw SQL, repository call) inside a loop body, comprehension, or callback applied to a collection.

### 2. Business Logic in Controllers / Views

**Project Example:** [discovered during init — e.g., "Found in `api/views/v1/manage/order_views.py:82`: `create_order` handler contains payment charge, inventory decrement, and notification logic" or "not yet found — will be flagged if detected"]
**Alternative:** [framework-specific — e.g., "Extract to `<Entity>Service.<action>()`. ViewSet only parses input → calls service → returns response." or "Extract to a Command Handler (MediatR/CQRS pattern)"]
**Detection:** A controller/view function longer than ~15 lines; any branching on domain rules; any call to an external service, payment gateway, or state machine from inside a controller.

### 3. Stack Traces in API Responses
**Project Example:** [discovered during init or "not yet found — will be flagged if detected"]
**Alternative:** Structured error payloads (RFC 7807 or project convention). Internal details logged server-side only; client gets correlation ID + human-readable message.
**Detection:** `str(exception)`/`repr(exception)` in response bodies; `traceback.format_exc()` near response; debug middleware in production; `error.message` with file paths.

### 4. Synchronous Long-Running Operations
**Project Example:** [discovered during init or "not yet found — will be flagged if detected"]
**Alternative:** [framework-specific — e.g., "Enqueue Celery task, return 202 + job ID" or "Dispatch to worker; return 202"]
**Detection:** Request handler calling a function >2s; `time.sleep()` in handler; loop over >1000 items inside a request.

### 5. Secrets Hardcoded in Source
**Project Example:** [discovered during init or "not yet found — will be flagged if detected"]
**Alternative:** 12-Factor Config — secrets from env vars, secret managers, or encrypted config. Source contains only placeholder references.
**Detection:** String literals with `password=`/`secret=`/`key=`/`token=` + non-placeholder value; URLs with embedded credentials; `sk_live_`/`AKIA` prefixes in source.

### 6. DEBUG Mode in Production
**Project Example:** [discovered during init or "not yet found — will be flagged if detected"]
**Alternative:** Env-based config. Debug from env var, always `False` in production.
**Detection:** `DEBUG = True` without env guard; `app.run(debug=True)` in production; dev middleware in production stack.

### 7. Unpaginated List Endpoints
**Project Example:** [discovered during init or "not yet found — will be flagged if detected"]
**Alternative:** [framework-specific — e.g., "paginated_response via CustomResponseMixin" or "DRF PageNumberPagination"]
**Detection:** `Model.objects.all()` returned directly; `Response(list(...))` without pagination; no `page`/`limit`/`offset`/`cursor` param.

### 8. Bare Catch-All Exception Swallowing
**Project Example:** [discovered during init or "not yet found — will be flagged if detected"]
**Alternative:** Catch specific exception types. Always log. Recover explicitly or let propagate as domain error.
**Detection:** `except Exception: pass`; bare `except:`; `except Exception as e:` where `e` is never logged/re-raised; empty catch blocks.

### 9. Direct State Mutation Outside Facade
**Project Example:** [discovered during init or "not yet found — will be flagged if detected"]
**Alternative:** Route every state change through the designated facade/service. It enforces valid transitions, triggers side effects, records history, maintains invariants.
**Detection:** Direct assignment to status/state/phase field outside entity's module; `entity.status = ...` in view/task/signal; `save()` on lifecycle fields without facade.

### 10. Missing Tenant/Context Filter on Scoped Data
**Project Example:** [discovered during init or "not yet found — will be flagged if detected"]
**Alternative:** Always scope queries by current tenant/context. Enforce at framework level so developers cannot forget.
**Detection:** `Model.objects.all()` on tenant-scoped model; queryset without `.filter(tenant=...)`; `get_queryset()` without tenant scoping; raw SQL without `WHERE tenant_id = ?`.

## How `/zoom-in init` populates this file

1. **§1 Universal No-Gos**: For each of the 10 patterns, the model searches
   the codebase for a concrete instance. If found, it records the file path
   and line number. The Alternative is filled with the framework-specific
   correct approach from the detected framework reference. If no instance is
   found, the placeholder text remains — it serves as a detection signal for
   future audits.

2. **§2 Framework-Specific Bans**: Extracted from the matching
   `references/frameworks/<detected>.md` file's "Framework Anti-Patterns"
   section. If the framework has no reference file, the section notes that
   universal rules still apply.

3. **§3 Register-Specific Bans**: Extracted from the matching
   `references/registers/<selected>.md` file's "Key Anti-Patterns" section.
   If multiple registers are selected, their tables are merged (deduplicated,
   higher severity wins). Default register produces no register-specific bans.

4. **§4 Project-Specific Bans**: Discovered by comparing golden reference
   patterns against patterns found in non-golden modules. Only patterns that
   golden references explicitly avoid qualify. If none are found, the
   placeholder text is left.

5. **§5 Team-Added Bans**: Always empty at init. Populated exclusively by
   `/zoom-in adopt --ban`.

## Relationship to other zoom-in files

| File | Scope | Updated by |
|---|---|---|
| `references/anti-patterns.md` | Universal — 10 No-Gos, framework-agnostic | Skill updates only |
| `ANTI-PATTERNS.md` (this template) | Project-specific — tailored examples + framework + register + team bans | `/zoom-in init`, `/zoom-in adopt --ban`, `/zoom-in audit` |
| `ARCHITECTURE.md` §4 | Known Exceptions — deviations that are allowed | `/zoom-in adopt`, `/zoom-in refactor` |

`references/anti-patterns.md` answers: "What patterns are ALWAYS wrong?"
`ANTI-PATTERNS.md` answers: "What patterns are wrong IN THIS PROJECT, and where have we seen them?"
`ARCHITECTURE.md` §4 answers: "What deviations are ALLOWED, and why?"

Every `/zoom-in audit` loads ANTI-PATTERNS.md alongside the universal reference.
Findings matching §1-§4 are `[principle]` with the severity listed above.
Findings matching §5 are `[principle]` at the severity specified when added.
No finding from this file is ever `[house-style]` — these are rejection rules, not preferences.
