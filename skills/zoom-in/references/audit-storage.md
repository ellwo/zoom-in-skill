# Audit Storage — Persisting Audit Results Across Sessions

> Audit results are machine-generated diagnostics. If they vanish when a session ends, every new audit starts from zero. Storage gives zoom-in memory: deltas, regressions, resolved findings — evidence of progress or decay.

---

## 1. Directory Structure

```
.zoom-in/
└── context/
    └── audits/
        ├── index.md                          # Master index
        ├── full/                             # Full project audits
        │   ├── 2026-06-14--score-42.md
        │   └── 2026-06-20--score-51.md
        ├── <module-name>/                    # Module-scoped audits
        │   └── 2026-06-15--score-38.md
        ├── @<lens-name>/                     # Lens-scoped audits
        │   └── 2026-06-16--score-6.md
        └── <path-with-dashes>/              # Path-scoped audits (slashes→dashes)
            └── 2026-06-17--score-35.md
```

- `.zoom-in/` — hidden directory at project root. Machine-managed, not hand-edited.
- `context/audits/` — all persisted audit results.
- **Scope → subdirectory mapping**:
  - `full` — full project audits
  - Module name — module-scoped (e.g., `billing/`)
  - `@lens-name` — lens-scoped (e.g., `@security/`)
  - Path with dashes — file/directory scope where `/` → `-` (e.g., `src-orders-services.md` for `src/orders/services.py`)
- **File naming**: `YYYY-MM-DD--score-XX.md` — date + total score for quick visual identification. Date uses the audit run date. Score is the total out of 70.

## 2. index.md — Master Index

```markdown
# Audit Index

> Auto-generated by /zoom-in audit. Do not edit manually.

## full
| Date | Score | 🔴 | 🟠 | ⚠️ | 💡 | Delta | File |
|---|---|---|---|---|---|---|---|
| 2026-06-14 | 42/70 | 3 | 5 | 8 | 4 | — | 2026-06-14--score-42.md |
| 2026-06-20 | 51/70 | 1 | 3 | 6 | 2 | +9 | 2026-06-20--score-51.md |

## billing
| Date | Score | 🔴 | 🟠 | ⚠️ | 💡 | Delta | File |
|---|---|---|---|---|---|---|---|
| 2026-06-15 | 38/70 | 2 | 4 | 5 | 3 | — | 2026-06-15--score-38.md |
```

- **Delta column**: change from previous audit in that scope. `—` for the first audit (no baseline).
- **Sort order**: date descending — newest first.
- **Purpose**: at a glance, rising scores = improvement, falling = regression. Severity counts show where pressure is concentrated.
- **One section per scope** — section heading matches the subdirectory name.

## 3. Audit File Format

Each audit file uses YAML frontmatter + markdown body:

```markdown
---
scope: full
date: 2026-06-20
score: 51
critical: 1
high: 3
medium: 6
low: 2
previous_score: 42
delta: +9
framework: django
registers: [saas, api-service]
audit_commit_sha: 266eabf6d48c0e2001a16f72db90f01c7f0773af
previous_audit_commit_sha: abc123def456
---

# Zoom-in Audit: full — 2026-06-20

## Score Progression
| Lens | Current | Previous | Delta |
|------|---------|----------|-------|
| Clarity | 8/10 | 7/10 | +1 |
| Structure | 7/10 | 5/10 | +2 |
| Performance | 9/10 | 8/10 | +1 |
| Security | 8/10 | 7/10 | +1 |
| Resilience | 9/10 | 7/10 | +2 |
| Domain Integrity | 10/10 | 8/10 | +2 |
| Observability | 6/10 | 4/10 | +2 |
| **Total** | **57/70** | **46/70** | **+11** |

## Findings by Severity
[... full audit report as defined in audit.md ...]

## Priority Actions
[... top 5 actions ...]

## Resolved from Previous Audit
[Findings from the previous audit that are no longer present — closed items]
- [SE-1] Missing input validation — ✅ Fixed (was 🔴 Critical)
- [ST-2] Circular dependency in billing — ✅ Fixed (was 🟠 High)

## New Findings (not in previous audit)
[Findings that appeared for the first time]
- [PF-3] Missing index on orders.created_at — 🟠 High
```

**Frontmatter** — enables quick scanning and tool parsing without reading the body. Fields:
- `scope`, `date`, `score` — identity
- `critical`, `high`, `medium`, `low` — severity distribution
- `previous_score`, `delta` — change tracking (absent on first audit)
- `framework`, `registers` — calibration context used
- `audit_commit_sha` — git HEAD commit SHA at audit time (`git rev-parse HEAD`).
  Used by Change Detection in the next audit to identify which files changed.
  Value is "non-git" if the project is not a git repository.
- `previous_audit_commit_sha` — the `audit_commit_sha` from the previous audit
  (absent on first audit). Enables `git diff <previous>..HEAD` for precise
  change detection between audits.

**Score Progression** — per-lens delta, not just total. A total improvement of +9 could mask a -3 regression in one lens offset by gains elsewhere. This table surfaces that.

**Findings by Severity** — the full audit report as produced by `/zoom-in audit` (see `audit.md` Step 6 format).

**Priority Actions** — top 5 actions from the audit, repeated here for self-contained readability.

**Resolved from Previous** — findings that existed in the previous audit but are absent in the current one. Celebrates fixes, proves progress, and documents what changed.

**New Findings** — findings present in the current audit that did not exist in the previous one. Highlights regressions or newly discovered issues. Not all new findings are regressions — some are discoveries the previous audit missed.

## 4. Auto-Cleanup Rule

**Keep only the 2 most recent audits per scope.**

Process after writing a new audit file:
1. List all files in the scope's subdirectory.
2. Sort by date (embedded in filename).
3. If more than 2 files exist: delete the oldest ones.
4. Update `index.md` to remove deleted entries.

**Why 2**: Current + previous is sufficient for delta comparison. More files accumulate noise — the value of a 5-audit-old report is near zero for delta purposes, and the file count grows indefinitely.

**Configurable depth**: Teams that want longer history can set `audit_history_depth: N` in SYSTEM.md. Default is 2. When present, the cleanup rule keeps N files instead of 2.

**Never delete the only audit in a scope.** Even if it's old, it's the only baseline. The minimum retained count is 1.

## 5. Delta Comparison Logic

When writing a new audit, compare with the previous one for the same scope:

1. Find the previous audit file for the same scope subdirectory.
2. Load its frontmatter + findings sections.
3. **Per-lens delta**: subtract previous lens score from current. Write to Score Progression table.
4. **Total delta**: subtract previous total from current. Write to frontmatter and index.md.
5. **Finding matching** — match findings by `(location + classification)`, not by ID. IDs may differ between audits. A finding at `path/to/file:42` classified as `[principle]` is likely the same issue across audits. Use this heuristic to categorize:
   - **Resolved**: finding exists in previous but not in current (same location + classification, no match in current).
   - **New**: finding exists in current but not in previous (no matching location + classification in previous).
   - **Persistent**: finding exists in both (same location + classification). Note if severity changed.
6. Write **Resolved from Previous** and **New Findings** sections.
7. Write delta values to frontmatter (`previous_score`, `delta`).

**No previous audit exists**: Skip delta sections. Omit `previous_score` and `delta` from frontmatter. First audit is a baseline — no comparison possible.

**Partial match**: If a finding's location changed (e.g., code moved to a new file) but description and classification match, treat as persistent with a note. Location-only matches with different classifications are not matches — they're independent findings.

## 6. Git Strategy

**Default: track in git.**

Rationale:
- Audit history is part of project health — like CI results or test coverage reports.
- Team visibility: everyone sees the architectural trajectory in `git log .zoom-in/context/audits/`.
- `git diff` on audit files shows exactly what changed between audits.

**Teams that prefer local-only**: Add `.zoom-in/` to `.gitignore`. The tool **never** modifies `.gitignore` — that is outside the tool's authority. The tool creates directories and writes files. Whether those files are tracked is the team's decision.

**The tool never modifies `.gitignore`.** Full stop.

## 7. Reading Audits

Other zoom-in commands consume persisted audits:

| Command | What it reads |
|---|---|
| `/zoom-in refactor` | Most recent audit for the target scope (full or specific). Falls back to conversation history if no persisted audit exists. |
| `/zoom-in init` | If `.zoom-in/context/audits/` exists, note previous audits. Ask: "Found previous audits. Use as baseline for pattern discovery?" |
| `/zoom-in audit` | Reads previous audit for delta comparison before writing the new one. |
| `/zoom-in plan` | Can reference previous audit's Priority Actions to know which areas still need attention. |

**Fallback behavior**: If no persisted audit exists for a scope, commands operate from conversation history or produce a fresh analysis. Persistence is an enhancement, not a requirement.

---

## 8. Directory Creation

When `/zoom-in audit` runs for the first time (or targets a new scope for the first time):

1. Create `.zoom-in/context/audits/` if it doesn't exist.
2. Create the scope subdirectory (e.g., `.zoom-in/context/audits/full/`).
3. Create `index.md` with the first entry.
4. Write the audit file.

No pre-setup required. No init command needed for storage. Directories and files are created on demand by the audit command itself.

**Atomic write**: Write the audit file first, then update `index.md`. If the audit file write fails, the index is not corrupted with a reference to a non-existent file.
