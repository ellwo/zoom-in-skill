# ARCHITECTURE.md Template

This template defines the structure of `ARCHITECTURE.md` at the project root.
When `/zoom-in init` generates the file, it follows this structure. When other
commands read it, they know where to find each piece of information.

---

> **This file is LIVING.** It gets updated by `/zoom-in init`, `/zoom-in adopt`, and
> grows with the project. It is NOT a one-time snapshot. Every adopted decision,
> every newly established pattern, and every resolved conflict should be
> reflected here.

## Status values for patterns

- **Established** — 3+ modules do it consistently; deviations are `[house-style]` findings
- **Emerging** — 1-2 modules do it; don't flag non-adopting modules, note the divergence
- **Inconsistent** — different modules do it differently with no clear winner; `[conflict]` finding
- **[to be discovered]** — not yet analyzed; `/zoom-in init` will fill this in

---

## Update rules

| Event | Section(s) affected | Action |
|-------|---------------------|--------|
| `/zoom-in init` | §1, §2, §5, §6 | Scan codebase, discover patterns, populate profile and tables |
| `/zoom-in adopt "<decision>"` | §3, §2 (if new pattern), §5 (if conflict resolved) | Add decision row; update status if pattern promoted |
| `/zoom-in audit` finds `[conflict]` | §5 | Add or update the Evolving Conventions row |
| `/zoom-in refactor` leaves unfixed finding | §4 | Add Known Exception with reason |
| Architecture changes over time | §1 | Update profile fields as stack/tenancy/auth evolves |

**Merging, not overwriting.** If `ARCHITECTURE.md` already exists, `/zoom-in init`
preserves §3 (Adopted Decisions) and §4 (Known Exceptions) in full. §2 is
updated only where new patterns emerged or statuses changed. §1 is updated
with new discovered context. §5 and §6 are rebuilt from the current scan.
